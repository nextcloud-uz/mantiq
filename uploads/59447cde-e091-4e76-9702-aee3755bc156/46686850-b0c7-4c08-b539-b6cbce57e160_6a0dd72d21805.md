## 1. What is a Self-Hosted Runner?

When GitHub Actions runs a job on `ubuntu-latest` , it uses a virtual machine on GitHub's servers. You pay nothing but you have no control over that machine — it resets completely after every job.

A **self-hosted runner** is different. It is a small agent program that you install on **your own machine** . When a pipeline job uses `runs-on: self-hosted` , GitHub sends that job to your machine instead of a cloud VM.

```
Normal job (GitHub-hosted)          Deploy job (self-hosted)
──────────────────────────          ────────────────────────
GitHub spins up a fresh VM          Job runs on YOUR laptop
Job runs there                      YOUR Docker gets the container
VM is deleted after the job         Container stays running on YOUR machine
```

### Why use self-hosted for deployment?

- The container runs **on your machine** — you can open `http://localhost:8080` in your browser
- No cost — you are using hardware you already have
- No network exposure — the runner connects outward to GitHub over HTTPS, nothing needs to be opened on your router

### How the connection works

```
Your machine                              GitHub
────────────────────────────────          ──────────────────────
runner agent ──── HTTPS polling ───►      Actions service
             ◄─── "new job ready" ───
             runs steps locally
             ──── log stream ──────►      Actions tab (live output)
```

The runner polls GitHub. GitHub never connects inward to you. No firewall rules or port-forwarding required.

## 3. Register the Runner — macOS

**Windows users:** skip to [Section 4](#4-register-the-runner--windows) .

### 3.1 Open the registration page on GitHub

1. Go to your repository on GitHub
2. Click **Settings** (gear icon, last tab in the top navigation)
3. In the left sidebar: scroll to **Code and automation** → click **Actions** → click **Runners**
4. Click the green **"New self-hosted runner"** button

### 3.2 Select your OS and chip

- Click **macOS**
- Architecture:
    - **ARM64** → Apple Silicon Mac (M1, M2, M3, M4)
    - **X64** → Intel Mac

Not sure which chip you have? Apple menu (top-left) → **"About This Mac"** — Says "Apple M…" → ARM64 — Says "Intel Core" → X64

### 3.3 Run the setup commands

GitHub shows you a set of commands with a **unique registration token** that expires in 1 hour. **Copy and run the commands from GitHub** , not from here — the token is different for every user and every session.

Open **Terminal** ( `Cmd + Space` → type `Terminal` → press Enter).

The commands look like this — use the ones GitHub shows you:

```
# 1. Create a dedicated folder for the runner
mkdir -p ~/actions-runner && cd ~/actions-runner

# 2. Download the runner package
#    (use the exact URL from the GitHub page — it includes your architecture)
curl -o actions-runner-osx-arm64-2.316.0.tar.gz -L \
  https://github.com/actions/runner/releases/download/v2.316.0/actions-runner-osx-arm64-2.316.0.tar.gz

# 3. Extract
tar xzf ./actions-runner-osx-arm64-2.316.0.tar.gz

# 4. Configure
#    (use the --token value from the GitHub page — it is unique to you)
./config.sh \
  --url https://github.com/YOUR_USERNAME/YOUR_REPO \
  --token AXXXXXXXXXXXXXXXXXXXXXXXXX
```

The configuration script asks a few questions. Press **Enter** for every question to accept the defaults:

```
Enter the name of the runner group:     [press Enter]
Enter the name for this runner:         [press Enter]  or type a name like "my-macbook"
Enter any additional labels:            [press Enter]
Enter name of work folder:              [press Enter]
```

You should see:

```
√ Runner successfully added
√ Runner connection is good
```

### 3.4 Start the runner

```
./run.sh
```

Expected output:

```
√ Connected to GitHub

Current runner version: '2.316.0'
Listening for Jobs
```

**Leave this terminal window open.** The runner stops when you close it. In [Section 8](#8-keep-the-runner-alive-after-reboot) we install it as a permanent service.

### 3.5 Confirm the runner appears on GitHub

Go back to the GitHub page (Settings → Runners) and refresh. You should see your machine listed with a **green dot** and the label **Idle** .

Green dot = connected and waiting for work. ✓

## 4. Register the Runner — Windows

**macOS users:** you already did Section 3, skip to [Section 5](#5-write-the-cross-platform-deploy-workflow) .

Same as Step 3.1 — repo → Settings → Actions → Runners → **"New self-hosted runner"** .

```
# 3. Extract
Expand-Archive -Path runner.zip -DestinationPath .

# 4. Configure
.\config.cmd `
  --url https://github.com/YOUR_USERNAME/YOUR_REPO `
  --token AXXXXXXXXXXXXXXXXXXXXXXXXX
```

Press **Enter** for all questions.

You should see:

```
√ Runner successfully added
√ Runner connection is good
```

### 4.5 Start the runner

```
.\run.cmd
```

Expected output:

```

Listening for Jobs
```

**Leave this PowerShell window open.**

## 5. Write the Cross-Platform Deploy Workflow

Now we add the **deploy job** to the pipeline. It runs on your machine, pulls the Docker image that was just pushed to Docker Hub, and starts a container.

### 5.1 Why Docker CLI is cross-platform

Docker commands ( `docker pull` , `docker run` , `docker rm` ) are identical on macOS and Windows. This means the deploy job itself does not need OS-specific branches — one set of commands works everywhere.

The only exception is the **health check** : macOS uses `curl` and Windows PowerShell uses `Invoke-WebRequest` . We handle this with `if: runner.os` on those two steps only.

### 5.2 Open `.github/workflows/ci.yml`

Add the `deploy` job below your existing `docker` job. You do not need to touch the `test` or `docker` jobs.

```
# ── Deploy job: pull image from Docker Hub and run as container ────────────
  deploy:
    name: Deploy to Local PC
    runs-on: self-hosted                 # runs on YOUR registered machine
    needs: docker                        # waits for the image to be pushed first
    if: github.ref == 'refs/heads/main' && github.event_name == 'push'
    environment: local-dev              # optional — add approval gate in Settings

    steps:

      # Log in so the runner can pull from Docker Hub
      - name: Log in to Docker Hub
        uses: docker/login-action@v3
        with:
          username: ${{ secrets.DOCKERHUB_USERNAME }}
          password: ${{ secrets.DOCKERHUB_TOKEN }}

      # Stop and remove the previous container if it exists
      # continue-on-error: true means this step is OK to fail on the very first
      # deploy when there is no container yet
      - name: Stop and remove existing container
        run: docker rm -f flask-demo
        continue-on-error: true

      # Pull the exact image version tied to this commit
      # Using github.sha instead of :latest guarantees we deploy exactly
      # what was built and tested in this pipeline run
      - name: Pull image from Docker Hub
        run: docker pull ${{ secrets.DOCKERHUB_USERNAME }}/flask-demo:${{ github.sha }}

      # Start a new container
      # -d            runs in the background (detached)
      # --name        gives the container a name so we can remove it next deploy
      # --restart     Docker auto-restarts the container if it crashes
      # -p 8080:8080  maps your machine's port 8080 to the container's port 8080
      - name: Run new container
        run: >
          docker run -d
          --name flask-demo
          --restart unless-stopped
          -p 8080:8080
          ${{ secrets.DOCKERHUB_USERNAME }}/flask-demo:${{ github.sha }}

      # Health check — macOS and Linux use curl
      - name: Health check (macOS / Linux)
        if: runner.os != 'Windows'
        run: |
          sleep 3
          curl -f http://localhost:8080/health
          echo "Deploy successful"

      # Health check — Windows PowerShell uses Invoke-WebRequest
      # (curl in PowerShell is an alias for Invoke-WebRequest with different syntax)
      - name: Health check (Windows)
        if: runner.os == 'Windows'
        shell: powershell
        run: |
          Start-Sleep 3
          Invoke-WebRequest -Uri http://localhost:8080/health -UseBasicParsing
          Write-Host "Deploy successful"
```

### 5.3 Full updated `ci.yml` from top to bottom

Here is the complete file in one block, ready to copy:

```
name: CI Pipeline

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

env:
  IMAGE_NAME: ${{ secrets.DOCKERHUB_USERNAME }}/flask-demo

jobs:

  test:
    name: Build & Test
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: "3.12"
      - uses: actions/cache@v4
        with:
          path: ~/.cache/pip
          key: pip-${{ hashFiles('requirements.txt') }}
      - run: pip install -r requirements.txt
      - run: pytest test_app.py -v

  docker:
    name: Docker Build & Push
    runs-on: ubuntu-latest
    needs: test
    if: github.ref == 'refs/heads/main' && github.event_name == 'push'
    steps:
      - uses: actions/checkout@v4
      - uses: docker/setup-buildx-action@v3
      - uses: docker/login-action@v3
        with:
          username: ${{ secrets.DOCKERHUB_USERNAME }}
          password: ${{ secrets.DOCKERHUB_TOKEN }}
      - uses: docker/build-push-action@v5
        with:
          context: .
          push: true
          tags: |
            ${{ env.IMAGE_NAME }}:latest
            ${{ env.IMAGE_NAME }}:${{ github.sha }}
          cache-from: type=gha
          cache-to: type=gha,mode=max

  deploy:
    name: Deploy to Local PC
    runs-on: self-hosted
    needs: docker
    if: github.ref == 'refs/heads/main' && github.event_name == 'push'
    environment: local-dev

    steps:
      - name: Log in to Docker Hub
        uses: docker/login-action@v3
        with:
          username: ${{ secrets.DOCKERHUB_USERNAME }}
          password: ${{ secrets.DOCKERHUB_TOKEN }}

      - name: Stop and remove existing container
        run: docker rm -f flask-demo
        continue-on-error: true

      - name: Pull image from Docker Hub
        run: docker pull ${{ env.IMAGE_NAME }}:${{ github.sha }}

      - name: Run new container
        run: docker run -d --name flask-demo --restart unless-stopped -p 8080:8080 ${{ env.IMAGE_NAME }}:${{ github.sha }}

      - name: Health check (macOS / Linux)
        if: runner.os != 'Windows'
        run: |
          sleep 3
          curl -f http://localhost:8080/health
          echo "Deploy successful"

      - name: Health check (Windows)
        if: runner.os == 'Windows'
        shell: powershell
        run: |
          Start-Sleep 3
          Invoke-WebRequest -Uri http://localhost:8080/health -UseBasicParsing
          Write-Host "Deploy successful"
```

## 6. Trigger a Real Deploy by Pushing a Commit

### 6.1 Before pushing — confirm both services are running

**Runner:** look at the terminal where you ran `./run.sh` (macOS) or `.\run.cmd` (Windows). It must show:

```
Listening for Jobs
```

If that window was closed, re-open it and start the runner again.

**Docker Desktop:** the whale icon in your menu bar or system tray must be steady (not spinning). If Docker is not running, start Docker Desktop and wait for "Engine running".

### 6.2 Commit and push the updated workflow

```
git add .github/workflows/ci.yml
git commit -m "ci: add self-hosted deploy job"
git push
```

### 6.3 Watch three things at once

As soon as the push completes, open these three things side by side:

**A — GitHub Actions tab** `https://github.com/YOUR_USERNAME/YOUR_REPO/actions` Click the new run. You will see three jobs in sequence:

```
┌────────────────┐     ┌──────────────────────┐     ┌──────────────────────┐
│  Build & Test  │ ──► │  Docker Build & Push  │ ──► │  Deploy to Local PC  │
│  GitHub VM     │     │  GitHub VM            │     │  YOUR machine        │
└────────────────┘     └──────────────────────┘     └──────────────────────┘
     ~1 min                   ~2 min                        ~20 sec
```

**B — Your runner terminal** When the Deploy job starts on GitHub, your runner terminal prints:

```
2024-01-15 10:23:01Z: Running job: Deploy to Local PC
```

Then each step's output appears live as it executes — exactly as if you typed those commands yourself.

**C — Docker Desktop** Open Docker Desktop → click **"Containers"** in the left sidebar. You will see `flask-demo` appear and change status from "Starting" to "Running".

### 6.4 Step-by-step: what your runner terminal shows

```
Running: docker/login-action@v3
  Login Succeeded

Running: docker rm -f flask-demo
  Error: No such container: flask-demo   ← OK on first deploy, step is set to continue-on-error

Running: docker pull johndoe/flask-demo:a3f5c8d9...
  a3f5c8d9: Pulling from johndoe/flask-demo
  Status: Downloaded newer image

Running: docker run -d --name flask-demo --restart unless-stopped -p 8080:8080 ...
  c7f3a1b2e4d5...   ← container ID printed by Docker

Running: Health check (macOS / Linux)
  {"status": "ok"}
  Deploy successful

Job succeeded.
```

### 6.5 All three jobs show green checkmarks

When you see ✅ on all three jobs the full pipeline is complete.

### 7.4 Make a code change and push again

To see the full loop in action, make any small change to `app.py` , commit, and push:

```

git add app.py
git commit -m "update: test the deploy loop"
git push
```

Watch the three jobs run again. After the deploy job finishes:

- `docker ps` shows a **new container ID** (the old container was removed and replaced)
- The browser reflects your code change

This is the GitOps loop: **code change → push → automatic deploy** .

## 8. Keep the Runner Alive After Reboot

Right now the runner stops if you close the terminal or restart your machine. Install it as a permanent background service so it starts automatically.

### macOS — launchd service

Stop the runner first ( `Ctrl + C` in the terminal), then:

```
cd ~/actions-runner

sudo ./svc.sh install
sudo ./svc.sh start
sudo ./svc.sh status   # confirm: Active: active (running)
```

Other commands:

```
sudo ./svc.sh stop       # temporarily stop
sudo ./svc.sh uninstall  # remove the service
```

### Windows — Windows Service

Close the PowerShell window where `.\run.cmd` is running, then open a **new PowerShell as Administrator** :

( `Win + X` → "Windows PowerShell (Admin)")

```
cd C:\actions-runner

.\svc.cmd install
.\svc.cmd start
.\svc.cmd status
```

Or check via the Services panel: `Win + R` → type `services.msc` → look for **"GitHub Actions Runner"** → Status: **Running**

Other commands:

```
.\svc.cmd stop       # temporarily stop
.\svc.cmd uninstall  # remove the service
```

### Also set Docker Desktop to start on boot

The runner service starts automatically, but it also needs Docker to be running.

**macOS:** Docker Desktop → Settings (gear icon) → General → check **"Start Docker Desktop when you log in"**

**Windows:** Docker Desktop → Settings → General → confirm **"Start Docker Desktop when you sign in to your computer"** is checked

## 9. Troubleshooting

### Runner shows "Offline" on the GitHub Runners page

The runner agent is not running on your machine.

**Fix:** open a terminal and start the runner again:

```
# macOS
cd ~/actions-runner && ./run.sh

# Windows PowerShell
cd C:\actions-runner; .\run.cmd
```

Or, if you installed it as a service, restart it:

```
sudo ./svc.sh restart        # macOS
.\svc.cmd restart            # Windows (Admin PowerShell)
```

### The deploy job is stuck in "Queued" and never starts

GitHub is waiting for a runner to pick up the job, but none is available.

**Check in order:**

1. Is the runner running? The terminal must show `Listening for Jobs`
2. Go to Settings → Runners — does your runner show a green dot or "Offline"?
3. Is Docker Desktop running? The runner starts fine but `docker` commands fail if Docker is not up

### "docker: command not found" in the deploy job

Docker is installed but the runner service cannot see it in the PATH.

**Most common cause on macOS:** the runner service was installed before Docker Desktop, so Docker's path ( `/usr/local/bin/docker` ) was not in the environment when the service registered.

**Fix:**

```
sudo ./svc.sh stop
sudo ./svc.sh uninstall

# Start Docker Desktop, confirm it is running, then reinstall:
sudo ./svc.sh install
sudo ./svc.sh start
```

**On Windows:** Docker Desktop may install `docker.exe` to a user-specific path that the system service does not inherit. Fix: reinstall Docker Desktop with "Add to PATH for all users" selected, then reinstall the runner service.

### "pull access denied" — cannot pull the image

The deploy job cannot authenticate with Docker Hub.

**Check in order:**

1. Did the Docker Build &amp; Push job (Job 2) succeed? If it failed, there is no image to pull
2. Are `DOCKERHUB_USERNAME` and `DOCKERHUB_TOKEN` set in GitHub Secrets? Settings → Secrets and variables → Actions
3. Does the token have **Read, Write, Delete** permissions? Docker Hub → Account Settings → Security → check the token

### "port is already allocated" — container cannot start

Something else is using port 8080.

Find what is using it:

```
# macOS
lsof -i :8080

# Windows PowerShell
Get-NetTCPConnection -LocalPort 8080
```

If it is a leftover container:

```
docker ps -a                  # list all containers including stopped
docker rm -f CONTAINER_NAME   # remove it
```

Then re-trigger the pipeline:

```
git commit --allow-empty -m "retrigger: free port 8080"
git push
```

### Health check fails: "curl: (7) Failed to connect to localhost"

The container started but the app crashed immediately.

Read the logs:

```
docker logs flask-demo
```

Common log errors and fixes:

`ModuleNotFoundError: No module named 'flask'` `Dockerfile` — the `RUN pip install` line must include flask | `Address already in use` `No such file or directory: '/app/app.py'` `COPY app.py .` line in the Dockerfile is missing or wrong |

| Error message   | Fix   |
|-----------------|-------|
|                 |       |

| Port 8080 is occupied. See "port is already allocated" above   |
|----------------------------------------------------------------|

### The `docker rm -f flask-demo` step shows an error but the pipeline continues

This is expected on the **first deploy** — there is no container to remove yet. The step has `continue-on-error: true` for exactly this reason. The error is safe to ignore.

### Runner registration token expired

The `--token` from the GitHub page expires after **1 hour** . If you see "Invalid token" during `./config.sh` , get a new one:

1. GitHub → Settings → Runners → New self-hosted runner
2. Copy the fresh token
3. Run `./config.sh --url ... --token NEW_TOKEN_HERE` again

*End of guide. Your machine is now a registered runner that automatically pulls and runs new Docker containers every time you push a commit to main.*
